## Железо и первые ограничения

Что было на руках:

| Компонент | Характеристики |
| --- | --- |
| NVIDIA Jetson Nano Dev Kit | 4 GB LPDDR4, ARM64, GPU Maxwell |
| Системный диск | microSD 64 GB |
| Хранилище данных | USB SSD 232 GB → 229 GB (ext4) |
| VPS | Ubuntu 24.04, 2 GB RAM, Европа |
| Роутер | Статический IP для Jetson в домашней сети |

Первое, что выяснилось при работе с Jetson Nano — нет swap-раздела. Это серьёзно ограничивает выбор: например, Zabbix требует отдельной PostgreSQL и 500+ MB RAM, OpenMediaVault захватывает систему целиком. Docker 20.10.7 — старый, обновить нетривиально: это ограничение платформы JetPack, от которого зависят все системные компоненты Jetson. ARM64 — не все образы имеют нативные сборки. Пришлось собирать гибридное решение из лёгких инструментов.

**Почему не Synology или QNAP?** У меня уже был Jetson — он достался «бесплатно» из коробки. Synology с сопоставимым функционалом стоит от 20 000 рублей, плюс подписки на пакеты. Здесь я получил контроль над каждой деталью — и заодно разобрался как это работает изнутри.

Отдельная история — внешнее хранилище. Но об этом — в разделе про USB-инциденты.

## Архитектура

### Доступ из интернета

Jetson стоит за домашним роутером в CGNAT — прямого входящего соединения нет. Решение — **reverse SSH tunnel** через VPS: Jetson сам устанавливает исходящее соединение, VPS проксирует трафик обратно.

```
[Смартфон / браузер / приложение]
         |
         | HTTPS / HTTP
         v
    [VPS в Европе]
    nginx (Docker)
    :8080 / :8443  ──┐
    :2283 / :2443  ──┤  reverse SSH tunnel (autossh)
    :8090 / :9443  ──┤
    :8099          ──┘
         |
         | SSH tunnel — исходящий от Jetson
         v
[Jetson Nano — домашняя сеть 192.168.x.x]
    Nextcloud    :8080   — файлы, фото, контакты
    Immich       :2283   — фотоархив
    LLM Gateway  :8090   — AI-ассистент
    NASA API     :8099   — управление и статистика
    Samba        :445    — только LAN
    Netdata      :19999  — метрики
    Uptime Kuma  :3001   — мониторинг
    Portainer    :9000   — Docker UI
         |
         | USB 3.0, 5 Gbps
         v
[JMS583 SSD 229 GB — /mnt/storage]
```

**Почему не WireGuard и не Tailscale?** WireGuard требует DKMS — несовместим с Tegra kernel 4.9. Tailscale конфликтует с уже работающим VPN на Android. autossh решает задачу без внешних зависимостей и работает стабильно через CGNAT с `Restart=always`.

**HTTPS без домена.** Let's Encrypt недоступен — нет доменного имени. Временное решение (пока нет домена): self-signed TLS на alt-портах (:8443, :2443, :9443) со сроком 10 лет. Браузер предупреждает один раз — принять сертификат. Следующий шаг — добавить домен и перейти на Caddy с автоматическим Let's Encrypt.

### 13 контейнеров на 4 GB RAM

Ключевые принципы при дизайне: `mem_limit` для каждого контейнера и `restart: always`. Без mem_limit один контейнер может занять всю память и уронить остальные.

| Контейнер | Образ | Порт | mem_limit |
| --- | --- | --- | --- |
| nextcloud | nextcloud:apache | 8080 | 512m |
| nextcloud_db | postgres:16-alpine | — | 512m |
| nextcloud_redis | redis:7-alpine | — | 64m |
| immich_server | immich-server:release | 2283 | 1024m |
| immich_db | pgvecto-rs:pg16 | — | 384m |
| immich_redis | redis:7-alpine | — | 64m |
| immich_microservices | immich-server:release | — | 512m |
| llm_gateway | custom FastAPI | 8090 | 256m |
| nasa_api | custom FastAPI | 8099 | 128m |
| samba | crazymax/samba | 445 | — |
| netdata | netdata:latest | 19999 | 256m |
| uptime_kuma | louislam/uptime-kuma | 3001 | 128m |
| portainer | portainer-ce | 9000 | 128m |

`IMMICH_DISABLE_MACHINE_LEARNING=true` — без этого флага Immich не запустится стабильно на 4 GB без swap.

## Три USB-инцидента, после которых проект стал инженерным

Это самая честная часть статьи. Хранилище данных — USB SSD — доставило три отдельных проблемы, каждая из которых могла закончить проект.

### Инцидент 1 — error -71 (RTL9210B-CG + autosuspend)

Первый энклоужер (чип RTL9210B-CG) работал нормально несколько дней, потом начал выдавать в dmesg:

```
usb 2-1.3: USB disconnect, device number X
sd 0:0:0:0: [sda] tag#0 FAILED Result: hostbyte=DID_ERROR...
```

Корень проблемы — два фактора вместе: RTL9210B-CG переходит в USB autosuspend при простое, и Tegra kernel 4.9 не умеет корректно его будить. Docker падал, данные были недоступны.

Попытки решить: `usbcore.autosuspend=-1` в загрузчике — помогло частично. Но RTL9210B-CG при записи через UAS постепенно деградировал до скорости 40 MB/s и начал выдавать stream errors. Решение — заменить энклоужер. Новый: **JMS583** (chip 152d:a583, USB 3.0, 5 Gbps).

### Инцидент 2 — сломанный USB-порт 4 на Jetson

После подключения JMS583 к порту 4 — та же симптоматика. Оказался физически нестабильный порт. Переткнул в **порт 2** — всё заработало. Теперь это зафиксировано в конфиге watchdog-сервиса (PORT=2) и в документации.

### Инцидент 3 — CRLF в shebang скриптов

После создания systemd-юнитов и bash-скриптов на Windows, они не запускались на Jetson. Причина: Windows сохраняет файлы с окончаниями строк `\r\n`, а bash видит `#!/bin/bash\r` — нераспознанный интерпретатор.

Симптом: `Failed to execute /usr/bin/env bash\r: No such file or directory`.

Решение — `.gitattributes` с `* text=auto eol=lf` для всего репозитория. После этого все скрипты конвертируются автоматически при git checkout на Linux.

### Итог по USB — что сейчас работает

```bash
# /boot/extlinux/extlinux.conf, добавить к строке APPEND:
usb-storage.quirks=152d:a583:u usbcore.autosuspend=-1
```

Флаг `u` переключает JMS583 в BOT mode вместо UAS. Без него скорость записи была 8 MB/s вместо 250. После quirk: Write **250 MB/s**, Read **172 MB/s**, ноль USB ошибок в dmesg.

Дополнительно: SCSI timeout поднят до 120 секунд (udev правило), watchdog-сервис делает power cycle при каждой загрузке, udev-триггер автоматически монтирует диск и поднимает все контейнеры если SSD переподключается в горячую.

## Как я работал с инструментами разработки

Основным инструментом разработки был Claude Code — AI-агент с доступом к файловой системе и SSH. Опишу конкретно как это работало, без лишних оценок.

### AGENTS.md — это не документ, это память

Один из главных инсайтов. `AGENTS.md` читается при каждом запуске новой сессии. Туда записываются жёсткие запреты, аппаратные ограничения, архитектурные решения. Например: «не трогать VPN-контейнеры на VPS — там 25 клиентов», «USB-порт 4 сломан, использовать только порт 2», «CRLF ломает bash скрипты — всегда сохранять LF».

Без этого файла каждый раз приходилось бы заново объяснять контекст. С ним — агент не повторяет уже зафиксированные ошибки.

### Где помощь работает

**Параллельная генерация однотипных файлов.** systemd-юниты, Docker Compose файлы, nginx-конфиги — когда нужно написать 5 похожих файлов с правильными mem_limit и healthcheck-ами, это быстрее через агента, чем вручную.

**Мониторинг выбора технологий.** Когда я хотел Zabbix — агент проанализировал 9 инструментов и объяснил почему Zabbix не подойдёт (нужна третья PostgreSQL, 500+ MB RAM — OOM на 4 GB без swap). Это сэкономило несколько часов.

**Документация.** ADR, CHANGELOG, двуязычные инструкции для семьи — это работа которую откладываешь вечно. С агентом делается параллельно с кодом.

### Где без человека не обойтись

**Специфика железа.** Агент не знает что USB-порт 4 физически сломан, что RTL9210B-CG нестабилен с kernel 4.9, что на VPS есть работающий VPN который нельзя трогать. Всё это нужно объяснять явно. Без этого контекста рекомендации обобщённые и иногда нерабочие. Первый совет по WireGuard был технически корректным в общем случае — но несовместимым с Tegra kernel.

**Финальная проверка.** Firewall, fstab, пароли — читать и верифицировать самому. Агент напишет конфиг, но то что он делает с вашей системой — ваша ответственность.

### Шаг 1 — Запуск базовых сервисов

Начал с Nextcloud и Immich. Промпт:

```
Подними Nextcloud и Immich на Jetson Nano.
PostgreSQL для обоих. Данные на /mnt/storage.
Учти: нет swap, 4 GB RAM.
```

Агент написал Docker Compose файлы с mem_limit и healthcheck-ами, а также добавил fail-closed логику в скрипт бэкапа: если `/mnt/storage` не смонтирован как отдельное устройство — бэкап не запускается. Это защита от случайного сохранения данных на microSD вместо SSD.

Проверял так:

docker compose ps
curl -s http://localhost:8080/status.php | python3 -m json.tool

### Шаг 3 — Автоматические тесты инфраструктуры

После каждого изменения хотел быть уверен, что ничего не сломал. Использовал goss — лёгкий инструмент проверки состояния инфраструктуры. Установка на ARM64:

40 тестов: порты открыты, systemd-сервисы активны, нужные файлы существуют, HTTP-эндпоинты отвечают. После любых изменений:

```
Count: 40, Failed: 0, Skipped: 0 — Duration: 4.32s
```

### Шаг 4 — HTTPS для Android-приложений

Выяснилось, что Immich на Android требует HTTPS для автобэкапа, а DAVx⁵ отказывается синхронизировать контакты по HTTP. Без доменного имени сделал self-signed сертификат и прописал IP в SAN:

```bash
openssl req -x509 -nodes -days 3650 -newkey rsa:2048 \
  -keyout /opt/nasa/nginx/ssl/nasa.key \
  -out /opt/nasa/nginx/ssl/nasa.crt \
  -subj "/CN=nasa-home-cloud" \
  -addext "subjectAltName=IP:ваш_VPS_IP, IP:192.168.x.x"
```

nginx на VPS добавил HTTPS-блоки на alt-портах. Приложение один раз спрашивает «доверять сертификату?» — после этого работает как обычно.

### Шаг 5 — Android-клиенты

Подключение семьи к системе потребовало учесть специфику телефонов Xiaomi с MIUI/HyperOS. Написал инструкцию с разделами про battery whitelist и автозапуск приложений — без этого Immich не делает бэкап в фоне.

Результат: **6 697 фотографий и видео** загружено с телефона. По данным Immich API: 6 484 фото + 210 видео (3 файла не приняты — повреждённые/дублированные, это норма). Автобэкап работает стабильно.

Контакты синхронизировал через DAVx⁵ — 2 151 запись, CalDAV и CardDAV.

## Безопасность: что закрыто

- **UFW на VPS**: разрешены только нужные порты (8080, 8443, 2283, 2443, 8090, 9443, 8099, 8091, 22). Всё остальное — DROP.
- **Amnezia VPN** на VPS изолирован — его контейнеры не затронуты проектом.
- **Нет открытых портов на Jetson** — только reverse tunnel, инициированный изнутри LAN.
- **Samba только в LAN** — не проброшен через туннель.
- **JWT авторизация** в NASA API — каждый запрос требует токен.
- **secrets-check перед каждым push** — `check_no_secrets.sh` в pre-commit.

Что не закрыто и я это знаю:

- Docker 20.10.7 содержит устаревшие компоненты с известными CVE. Обновление ограничено JetPack — это платформенный constraint, не забывчивость.
- Self-signed TLS — браузер предупреждает. Временное решение до появления домена.

## Честная оценка

**Что ещё не сделано (честно):**

- Docker 20.10.7 устарел — обновить нетривиально из-за JetPack зависимостей. Риск: CVE в старых образах.
- Off-site backup не настроен — скрипты restic готовы, развёртывание планируется после подключения второго 2 ТБ HDD. До тех пор данные есть только на одном диске — это риск.
- ML в Immich отключён — Jetson 4 GB без swap не тянет распознавание лиц и объектов.
- Self-signed TLS — временно. Когда появится домен, перейду на Caddy + Let's Encrypt.

## Что бы я сделал иначе

**Сразу бы купил правильный энклоужер.** Два инцидента с USB — это потеря нескольких дней и риск потери данных. JMS583 стоит не дороже DEXP-бокса с RTL9210B-CG, но работает стабильно с Tegra kernel из коробки (после одной строки quirk).

**Сразу .gitattributes с `eol=lf`.** CRLF-проблема убила час и убивала бы ещё — если бы не зафиксировал в первый же день работы с Windows.

**Сразу AGENTS.md.** Я написал его не сразу — первые несколько сессий тратил время на повторное объяснение контекста. Этот файл нужен с первой строки проекта.

**Домен сразу, не потом.** Self-signed TLS — не проблема дома, но усложняет onboarding новых пользователей. Дешёвый домен (.ru или .xyz) с Caddy решил бы это раз и навсегда.
