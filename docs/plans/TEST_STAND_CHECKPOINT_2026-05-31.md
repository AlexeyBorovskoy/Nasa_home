# Тестовый стенд NASA — чекпойнт и разбор сбоя (2026-05-31)

> Точка фиксации проекта после сессии развёртывания тестового стенда на реальном
> Jetson Nano. Особое внимание — разбору сбоя пользовательского VPN (§4.2).
> Секретов в этом файле нет (все реквизиты — в `config/.env` и `.master.env`).

## 1. Резюме

Развёрнут **тестовый стенд** на реальном Jetson Nano (без HDD — на временной
USB-флешке). Работают **LLM Gateway** и **Nextcloud**. Jetson подключён к
домашнему роутеру по LAN. Внешний VPN-доступ **не завершён**: две попытки
(самодельный WireGuard и AmneziaWG) не доведены; вторая **вызвала сбой
пользовательского VPN** (кратковременно пропал интернет на телефоне). Все
рискованные изменения **откачены**.

## 3. Сетевой путь (как шли)

1. Первичная настройка Jetson — офлайн правкой rootfs (см. `docs/01A` §11.2), т.к.
   USB-serial консоли на стоке нет, монитора не было.
2. Контроль и интернет — через **USB device-mode** (`192.168.55.1`) + временный
   NAT с ноутбука.
3. Перенос Jetson на **домашний роутер** (eth0 → LAN-порт TP-Link), статика
   `192.168.0.50`.
4. Дом за **CGNAT** (WAN из диапазона `100.64.0.0/10`) — **прямой проброс портов
   невозможен**, только исходящие VPN-схемы.

## 4. СБОЙ — разбор (главное)

### 4.1. Самодельный WireGuard (`wg-nasa`) через CGNAT — нестабилен

Подняли отдельный wg-интерфейс на EU VPS (`45.95.2.49:51820`, подсеть
`10.13.13.0/24`), Jetson-пир; ядро 4.9 → модуль собран через `wireguard-dkms`.
Хендшейк вставал, но **ICMP проходил с перебоями, а TCP не проходил вообще**
(SYN уходил с VPS — на Jetson не приходил; снижение MTU до 1280 и MSS-clamp не
помогли). **Вывод:** самодельный WG из-за CGNAT нестабилен. **Удалён полностью.**

### 4.2. AmneziaWG-клиент — ВЫЗВАЛ СБОЙ пользовательского VPN ⚠️

Решили использовать рабочую Amnezia-инфраструктуру на том же VPS. Собрали
`amneziawg-dkms` под ядро 4.9, поставили `amneziawg-tools`, подготовили
клиентский конфиг (`10.8.1.26` с параметрами обфускации сервера).

**Сбой:** добавление пира на сервер Amnezia командой `wg set` (вживую, внутри
контейнера `amnezia-awg`) **спровоцировало авто-reconcile Amnezia с рестартом
контейнера**. На момент рестарта **отвалились ВСЕ ~25 VPN-клиентов, включая
телефон пользователя → у пользователя кратковременно пропал интернет.**

**Корневая причина:** Amnezia сама управляет своим конфигом
(`/opt/amnezia/awg/wg0.conf`) и **откатывает любые внешние правки с рестартом**.
Наш тестовый пир при рестарте был удалён (вернулось 25 пиров) — следов не осталось.

**УРОК (важно):** **никогда не править конфиг работающего Amnezia-сервера через
SSH / `wg set`.** Добавлять клиентов только штатно — через десктоп-приложение
Amnezia (или его API), которое делает это без рестарта и не роняет других клиентов.

### 4.3. Сопутствующие проблемы

- Временный дефолт-маршрут Jetson через ноутбук (`default via 192.168.55.100`)
  становился «мёртвым шлюзом» при отключении USB → Jetson терял интернет, пока
  маршрут не убрать. Урок: не оставлять временные маршруты через нестабильное
  плечо; полагаться на маршрут роутера.
- После **ручного ребута VPS** `wg-yandex` (туннель Amnezia↔Yandex VM) **не
  поднялся** — он не в автозапуске. Это **не наша** настройка; восстановить
  отдельно при необходимости.

## 5. Откат (выполнено на 2026-05-31)

- **VPS:** `wg-quick@wg-nasa` остановлен и снят с автозапуска; удалены
  `/etc/wireguard/wg-nasa.conf` и все ключи `nasa_*`; вычищены iptables
  (FORWARD `amn0↔wg-nasa`, MSS-clamp). **Amnezia цела** (контейнер Up, 25 пиров,
  NAT `172.29.172.0/24` на месте).
- **Ноутбук (VM):** удалены добавленные NAT-правила (`MASQUERADE -o ens33`,
  FORWARD для device-mode интерфейса). `ip_forward` не трогали (нужен docker).
- **Jetson:** на момент чекпойнта **недоступен** (USB не проброшен в VM, VPN
  удалён, LAN из VM не виден). Откат сетевых настроек Jetson **НЕ выполнен** —
  ожидает доступа по USB или физического отключения.

## 6. Что осталось сделать

1. **Откатить сетевые настройки на самом Jetson** (когда будет доступ по USB):
   - удалить `/etc/wireguard/wg-nasa.conf` + `wg-quick@wg-nasa` disable;
   - удалить `/etc/amneziawg/awg0.conf` (туннель не поднимался);
   - вернуть eth0 в исходное (удалить NM-профиль `nasa-lan`, включить DHCP-профиль
     `Wired connection 1`), убрать ручные маршруты, восстановить `/etc/resolv.conf`.
2. **`wg-yandex`** на VPS — решить, восстанавливать ли туннель пользователя.
3. **Внешний доступ к стенду** — выбрать способ, НЕ трогающий Amnezia-сервер:
   - **Tailscale на Jetson** (рекомендуется: userspace-WG, пробивает CGNAT, ноль
     риска для рабочего VPN), либо
   - добавить клиента **через десктоп-приложение Amnezia** (штатно, без рестарта).
4. **HDD:** при появлении — миграция `/mnt/storage` с временной флешки на HDD.
